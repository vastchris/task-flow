
# 3. Markdown 文档格式规范

## 3.1 基本文档结构

示例：

```md
# 2026.6.1-6.7
%% week:2026.6.1-6.7 %%
- [ ] 小论文 ^tf-w-001
	- [ ] 查资料 ^tf-w-002
	- [ ] 写摘要 ^tf-w-003
%% week end %%

## 2026.6.1
%% day:2026.6.1 %%
- [ ] 小论文 ^tf-d-101
	- [/] 查资料 ^tf-d-102
%% day end %%

### 查资料 / 小论文
tasklog:: tf-d-102
***
这里写工作记录。
```

插件只识别：

```md
%% week:... %%
...
%% week end %%

%% day:... %%
...
%% day end %%

tasklog:: tf-d-xxx

```

普通正文里的 checklist 不接管。

## 3.2 任务格式与状态

任务使用 Markdown checklist。

主状态：

```md
- [ ] 未开始
- [/] 进行中
- [x] 已完成
```

特殊状态：

```md
- [ ] ✅ 任务名
- [/] ✅ 任务名
```

含义：

```text
[ ] ✅ = 这天计划了，但当天没开始，后来别的日期完成
[/] ✅ = 这天做了一部分，后来别的日期完成
```

侧边栏对应显示：

```text
[ ]      -> 红圈
[/]      -> 蓝圈
[x]      -> 绿色圆形勾
[ ] ✅   -> 灰圈 + 小绿角标
[/] ✅   -> 蓝圈 + 小绿角标
```

## 3.3 父子任务格式

支持一级父子任务：
父子关系以 data.json 为主，Markdown 任务行保持同样层级用于文档同步。
```md
- [ ] 项目 A ^tf-w-001
	- [ ] task A.1 ^tf-w-002
	- [ ] task A.2 ^tf-w-003
```

规则：

```text
tasklog 只绑定没有子任务的 Day 任务，包括 Day 子任务和 Day 独立任务。父任务和 Week 任务没有 tasklog。
```

## 3.4 任务 ID

任务使用 Obsidian block ID：

```md
- [ ] 查资料 ^tf-d-102
```

ID 规则：

```text
ID 由插件自动生成
已有 ID 不重复生成
每个任务 ID 唯一
同名称任务也不共用 ID
```

ID生成
用户在侧边栏成功创建任务之后任务对应的id就创建好

ID 格式：

```text
周任务：tf-w-xxxx
日任务：tf-d-xxxx
```

## 3.5 工作记录格式

工作记录是普通 Markdown，不放进代码块。

格式：

```md
### 当前任务 / 父任务
tasklog:: tf-d-102
***
这里写真实工作记录。
```

字段含义：

```text
tasklog:: 绑定的 day 任务 ID
```

插件通过 200ms 防抖扫描当前文档中的 `tasklog:: tf-d-xxx`，判断某个 Day 非父任务是否已有工作记录。

活动编辑器中的状态同步使用 Editor API 只替换任务状态行，不整篇覆写文件，以避免和 Obsidian Sync 冲突。
## 3.6 侧边栏内容添加进md文档

对于阶段4的任务创建来说
需要在文档中对应的区域去增加上任务行
如果发现文档现在内容里面没有对应的格式区域的话弹出提示让用户去自己创建一下。
对于阶段5的任务删除来说
需要在文档中对应的区域去删除对应的任务
对于阶段6的任务修改来说
需要在文档中把这个任务修改，顺序也一样
对于日期移动的话，如果没有文档中没有对应的日期的区域格式的话也要弹出提示请在文档中创建对应区域
就是相当于这些操作在侧边栏去做的时候都得在文档中先有对应的格式区域才能在侧边栏进行。然后操作完之后再在对应的区域去更新文档内容

状态修改由插件计算后同步到对应任务行。tasklog 仍由用户点击红圈复制模板后手动粘贴；粘贴或删除 tasklog 会自动更新任务、同源组、父任务和 Week 状态。
