# 7. 任务状态变更与联动

## 当前实现摘要

- 状态源头只来自 Day 非父任务
- 跨日期同源实例按一个身份组计算
- Week/Day 父任务按子任务身份组汇总状态和进度
- tasklog 变化采用 200ms 防抖和单轮批量计算
- 活动编辑器使用 Editor API 行级写回，非活动文件才使用 Vault
- 结构变更完成后执行全局状态重算

任务状态更新分为两类：

```text
1. 常规状态改变
	由侧边栏圆圈点击、tasklog 新增 / 删除触发。

2. 结构变化后的全局状态重算
	由前面的任务结构更新完成后（任务创建，任务删除，任务修改）自动触发。
```

## 7.0 任务状态
侧边栏显示：
```text
md文档    | 侧边栏
[ ]      -> 红圈
[/]      -> 蓝圈（内部填充浅蓝色）
[x]      -> 绿色圆形勾
[ ] ✅   -> 灰圈 + 小绿角标
[/] ✅   -> 蓝圈 + 小绿角标
```

基本状态：
```md
- [ ] 未开始
- [/] 进行中
- [x] 已完成
```

特殊状态：
```md
- [ ] ✅ 任务名
- [/] ✅ 任务名
```

含义：
```text
[ ] ✅ = 这天计划了，但当天没开始，后来别的日期完成
[/] ✅ = 这天做了一部分，后来别的日期完成
```

---
## 7.1 常规状态改变

### 7.1.1 常规状态变更触发入口

状态变更只由两类入口触发：

```text
1. 侧边栏圆圈点击
2. tasklog 集合变化
```

任务结构更新不属于常规状态变更入口。
任务结构更新：
	完成后触发 7.2 全局状态重算，不直接走 7.1 常规状态改变。

#### 7.1.1.1 圆圈点击

点击红圈：

```text
点击侧边栏任务的红圈后解析得到该任务的id，从data.json数据中获取该id任务的如下信息：
	任务名称
	父任务名称（若存在父任务）
	任务id
通过这些信息得到工作记录模板

具体复制内容就是3.5工作记录格式对应的内容，这里简单示例一下：
	### 任务名称 / 父任务名称
	tasklog:: tf-d-102
	***
	这里写真实工作记录。

注意：
点击红圈只复制工作记录模板。
不直接改变任务状态。
```

点击蓝圈：

```text
对应id的任务状态变化：
[/] -> [x]
```

点击绿勾：

```text
状态变化：
[x] -> [/]
```

注意圆圈变为：
- 灰圈 + 小绿角标 （`[ ] ✅ = 这天计划了，但当天没开始，后来别的日期完成`）
- 蓝圈 + 小绿角标 （`[/] ✅ = 这天做了一部分，后来别的日期完成`）
这两种圆圈是不能点击的了，或者说点击之后啥也不干

父任务和week任务的圆圈不可点击，只显示状态和进度。

注意：

```text
[x] 必须来自用户点击完成。
tasklog 只能让任务进入 [/]
不能自动让任务进入 [x]
```

#### 7.1.1.2 tasklog 变化

任务开始和取消开始由 tasklog 决定。
而且tasklog只有非父 day 任务才会有
规则：

```text
文档中新增 tasklog:: tf-d-xxx
对应任务从 [ ] 变成 [/]

文档中删除 tasklog:: tf-d-xxx
对应任务从 [/] 或 [x] 回到 [ ]
```

检测方式：

```text
200ms 防抖后的全文轻量扫描
```

说明：

```text
活动 Markdown 编辑器优先读取 Editor 当前文本；没有活动编辑器时读取 Vault 文本。
当文档发生编辑后等待 200ms。
扫描当前 Markdown 文档中的 tasklog:: 集合。
与上一次 tasklog 集合对比。
只有发现 tasklog 集合新增或删除时，才触发状态更新。
同一轮新增和删除批量计算，只生成一次状态写回结果。
```

扫描对象仅为 Markdown 文本，不读取图片附件内容。
***
通过对圆圈点击和tasklog的详细说明，可以得知：
其实任务状态更新其实源头变化只来自于非父 day 任务
因为上面的两条原则：
- 父任务和week任务的圆圈不可点击，只跟随更新状态和进度。
- tasklog只有非父 day 任务才会有
所以下面的内容就从非父 day 任务状态的变化规则来开始
### 7.1.2 非父 day 任务状态规则

非父 day 任务的定义：
没有子任务的日任务（这个日任务可以是day创建的也可以是有week来源的）

非父 day 任务的基础状态由 tasklog 和点击圆圈共同决定。

```text
没有 tasklog -> [ ]
有 tasklog，且用户未点完成 -> [/]
有 tasklog，且用户点了完成 -> [x]
```

注意：

```text
[x] 不是单靠 tasklog 自动产生的。
[x] 必须来自用户点击完成。
```

如果 tasklog 被删除：

```text
[/] -> [ ]
[x] -> [ ]
```

### 7.1.3 跨日期同源 day 非父任务状态
跨日期同源 day 非父任务，它的定义是：
- 属于日任务下，没有子任务的任务中，来源相同的任务

判断逻辑可以是：
找到属于日任务的任务
	在这些任务中找到没有子任务的任务
		进一步在这些任务中找来源相同的任务

来源相同后面用同源来表达

同源判断：

```text
有 sourceWeekTaskId：
	以 sourceWeekTaskId 作为同源 key

没有 sourceWeekTaskId 但有 sourceDayTaskId：
	以 sourceDayTaskId 作为同源 key

sourceWeekTaskId 和 sourceDayTaskId 都没有：
	只代表自己
```

跨日期同源 day 非父任务的同源组状态规则：

```text
如果存在 [x]：
	整体已完成
	最新的 [x] 是完成实例（按日期最晚的为准；同一天多个 [x] 取 ID 靠后的）
	其他任务状态变化：
		任务原来是 [ ] -> [ ] ✅
		任务原来是 [/] -> [/] ✅
		完成当天任务保持 [x]

	
如果不存在 [x]，但存在 [/]：
	整体进行中
	对于[ ] ✅和[/] ✅的状态去掉✅
	其他任务状态都保持原样

	
如果全部是 [ ]：
	整体未开始
```

补充说明：

```text
同源任务曾经被延续，不代表它之后永远必须保持进行中。
如果用户删除了该同源组所有实例的 tasklog，全部实例可以回到 [ ]。
这些跨日期实例继续保留，作为原有日期安排记录；状态和父任务进度仍将它们合并为一个同源任务计算。
插件不因 tasklog 删除自动删除或合并任务实例，避免隐式结构变更。
当同源组最后一个 tasklog 被删除时，侧边栏提示：

任务 A 的所有工作记录均已删除，跨日期实例（6.8，6.9...）已保留并全部恢复为未开始。
```

### 7.1.4 day父任务状态与进度
注意这里面讨论的都是day区域的父任务

父任务表示一个任务组的整体状态。

### 7.1.4.1 子任务身份组

父任务不是按 Day 实例数量统计，而是按“子任务身份组”统计。

```text
子任务身份组 = 代表同一项实际工作的全部子任务实例

只存在一个日期：
	单个实例构成一个身份组

来自同一个 Week 子任务：
	不同日期的 Day 实例合并为一个身份组

Day 子任务延续：
	延续根和全部延续实例合并为一个身份组
```

父任务进度：

```text
完成身份组数 / 身份组总数
```

**父任务状态基础规则：**

```text
所有子任务未开始 -> 父任务 [ ]

部分子任务进行中或完成 -> 父任务 [/]

所有子任务完成 -> 父任务 [x]
```

身份组有效状态：

```text
组内存在 [x] → 已完成
不存在 [x]，但存在 [/] → 进行中
全部 [ ] → 未开始
```

如果这个day父任务，它只存在一个日期下。那就直接用常规逻辑即可。
相反，如果这个day父任务后面还在多个日期中有出现，那就看下面的详细规则（本质上就是基本规则的基础上加了一些预处理）
这个判断的话要把day父任务分成两类：week生成的day父任务和day创建的day父任务。
1.对于week生成的day父任务
它判断这个day父任务是否只存在一个日期，就看它的week源任务的weektdayid字段里面是不是只有它这一个Id。
2.对于day创建的day父任务
它判断这个day父任务是否只存在一个日期，就看它的数据字段里面`"sourceDayTaskId"和"daytdayTaskIds"`是不是都为null，只要有一个不是null，说明他是延续日任务或者延续根任务。那就说明这个day父任务绝对在这周内多天出现了。

**父任务状态详细规则**

1. 找到同一个逻辑父任务在所有日期下的真实父实例。
2. 收集这些父实例的全部子任务。
3. 按子任务身份分组，跨日期同源实例只保留一个统计单位。
4. 先计算每个身份组的有效状态，再汇总父任务状态和进度。
5. 同一个逻辑父任务的所有 Day 实例显示相同状态和总进度。

### 7.1.5 父任务进度详情

父任务行统一显示：

```text
状态圆圈 + 任务名称 + 总进度 x/n + 加号 + 悬停拖拽手柄
```

进度展开后按身份组分为：

```text
进行中
未开始
已完成
```

跨日期身份组只显示一行：

```text
最新日期 + 状态圆圈 + 任务名称
其他日期：日期列表
```


### 7.1.6 week / day任务同步原则

Week区域的每一个任务，根据它是否产生了Day实例，分别按以下规则同步状态：

**Week父任务（有Day实例的）**
Day父实例的状态和进度在7.1.4已经算好了。Week父直接取值：
- 状态 = Day父实例的汇总状态
- 进度 = Day父实例的总进度（已完成子数 / 子任务总数）

**Week父任务（没有Day实例的）**
子任务没有被安排到Day（否则父也会跟着去Day）。直接用Week侧子任务的自身状态，按父任务基础规则汇总。

**Week子任务或独立任务（有Day实例的）**
到Day侧找对应的Day实例：
- Day实例有同源组 → 取7.1.3合并后的同源组状态
- Day实例没有同源组（只在一个日期出现过） → 取该Day实例的自身状态

**Week子任务或独立任务（没有Day实例的）**
不受Day影响，保持自身状态不变。

注意：以上同步方向是 Day → Week。Week任务的状态永远从Day派生，不会反过来。
### 7.1.7 常规状态更新总流程

无论是圆圈点击，还是 tasklog 变化，最终进入统一流程。

```text
0. 圆圈点击 / tasklog 变化触发出现

1. 读取当前 Markdown 中的tasklog 集合

2. 读取data.json

3. 应用本次触发带来的 day 非父任务状态变化
	圆圈点击：
		[/] -> [x]
		[x] -> [/]
	tasklog 新增：
		[ ] -> [/]
	tasklog 删除：
		[/] 或 [x] -> [ ]

4. 计算这个状态改变的 day任务的跨日期同源 day 非父任务状态

5. 判断这个变化的day任务和它的跨日期同源 day 非父任务。有没有父任务
	有父任务的话，
		就去按照7.1.4父任务状态与进度的规则，更新父任务状态和进度，再继续下面的第6步
	没有的话
		直接继续第6步
		
6. 把day区域的更新同步到week中，父子都对应同步就行

7. 把计算后的新状态在侧边栏显示，并自动更新md文档对应区域的内容
```

### 7.1.8 Day 移动与延续入口

本规则只适用于 Day 区域：

| 当前日期实际任务状态 | 右键入口 |
|--------------------|---------|
| 全部 `todo` | 只显示移动 |
| 全部 `doing` | 只显示延续 |
| 状态混合或包含 `done` | 移动和延续均隐藏，并显示提示 |

父任务按当前日期旗下全部子任务判断。多选时先展开选中的父任务，再按实际任务集合统一判断。Week 区域不显示 Day 移动或延续入口。

## 7.2 结构变化后的全局状态重算

结构变化后的全局状态重算，是第 6 章任务结构更新完成后自动触发的状态计算流程。

它和 7.1 使用同一套状态规则，但不是 7.1 的单点联动流程。

```text
7.1 常规状态改变：
	知道具体是哪个非父 day 任务发生状态变化。
	从这个任务出发，沿着同源关系、父子关系、week/day 关系做局部联动。

7.2 全局状态重算：
	不关心具体哪个任务刚刚变化。
	根据当前 Markdown、和最新的data.json对当前文档内任务状态进行全局分组汇总。
```

### 7.2.1 触发时机

任务结构更新完成后，自动触发一次全局状态重算。

也就是说，只要任务结构更新流程完成，就执行 7.2。



## 7.3 md 文档同步

### 7.3.1 流程顺序

状态操作不需要前置区块检查（不涉及任务结构变化，只改已有行的标记）。

```
1. 圆圈点击 / tasklog 变化触发
2. → 7.1 或 7.2 状态计算
3. → store.mutate()：更新任务状态
4. → 刷新侧边栏
5. → 活动编辑器使用 Editor API 替换状态行；非活动文件使用 Vault 写回
```

### 7.3.2 状态标记写入

按 `^任务ID` 定位任务行，替换 `[ ]` / `[/]` / `[x]` 标记：

| 侧边栏操作 | md 文档变化 |
|-----------|------------|
| 蓝圈点击 | `[/]` → `[x]` |
| 绿勾点击 | `[x]` → `[/]` |
| tasklog 新增 | `[ ]` → `[/]` |
| tasklog 删除 | `[/]` 或 `[x]` → `[ ]` |
| 同源组合并计算 | `[ ]` → `[ ] ✅` 或 `[/]` → `[/] ✅` |
| 同源组取消合并 | `[ ] ✅` → `[ ]` 或 `[/] ✅` → `[/]` |

只写回状态或特殊角标发生变化的行，不做全文重写。

活动 Markdown 编辑器必须使用 `Editor.replaceRange()` 修改对应任务行，不能使用 `vault.modify()` 整篇覆写，否则会与编辑器未保存缓冲区和 Obsidian Sync 形成双写冲突。

### 7.3.3 红圈点击 — tasklog 模板复制

红圈点击不写入文档。只生成模板复制到剪贴板：

```
1. 用户点击红圈
2. → 解析任务 ID → 从 data.json 获取任务名称、父任务名称
3. → 生成模板：

### 子任务名称 / 父任务名称
tasklog:: tf-d-xxxx
***

4. → 写入系统剪贴板
5. → Notice("工作记录模板已复制到剪贴板")
```

模板由用户手动粘贴到文档中合适位置，插件不自动插入。

### 7.3.4 tasklog 变化扫描

200ms 防抖扫描当前 Markdown 全文，提取 `tasklog:: xx` 集合：

- 正则匹配：`/tasklog::\s*(tf-d-\d+)/g`
- 与上一次扫描结果对比，检测新增和删除
- 新增 → 触发状态更新（`[ ]` → `[/]`）
- 删除 → 触发状态更新（`[/]` 或 `[x]` → `[ ]`）
- 不包含 `tasklog-removed::`
- 同一轮所有新增和删除批量计算
- 活动编辑器通过行级 Editor 修改写回；非活动文件才使用 Vault

### 7.3.5 tasklog 解绑

删除任务导致的解绑（与阶段 5 的 9.3 节对应）：

```
tasklog:: tf-d-xxx  →  tasklog-removed:: tf-d-xxx
```

工作记录正文保留不变。插件直接修改文档，不需要用户手动操作。

## 7.4 全局状态重算详细流程

```text
1. 读取更新后的data.json。

2. 扫描当前 Markdown 中的 tasklog:: 记录，得到 currentTaskLogs。
   currentTaskLogs 只包含当前仍然存在的有效 tasklog 绑定。
   不包含 tasklog-removed::。

3. 对比data.json中的 confirmedTaskLogs 和 currentTaskLogs，只找出当前 Markdown 中已经不存在的 tasklog：

   removedTaskLogIds = confirmedTaskLogs - currentTaskLogs

   也就是说：
   	confirmedTaskLogs 中有
   	currentTaskLogs 中没有
   	=> 说明这个 tasklog 已经被删除或解除绑定

4. 对 removedTaskLogIds 中仍然存在于当前 Markdown 的非父 day 任务，重新确定基础状态：

   如果该任务当前是 [/] 或 [x]：
   	改为 [ ]

   如果该任务当前是 [ ]、[ ] ✅、[/] ✅：
   	基础状态按 [ ] 处理

   如果 removedTaskLogIds 中的某个 taskId 对应任务已经不存在：
   	不写回任务状态
   	只在最后更新 confirmedTaskLogs
   	
5. 对所有跨日期同源 day 非父任务组，按照 7.1.3 的规则整体更新状态。

6. 对所有day父任务，按照 7.1.4 的规则整体更新状态。

7. 对应更新week区域的任务状态

8. 用 currentTaskLogs 覆盖 confirmedTaskLogs。

9. 更新侧边栏。
   
10. 将计算后的状态和进度与当前 Markdown 对比。
    只写回状态或进度发生变化的任务行。
```
