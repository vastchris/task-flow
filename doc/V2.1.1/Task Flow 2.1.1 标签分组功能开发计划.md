# Task Flow 2.1.1 标签分组功能开发计划

## 目标

基于 `Task Flow 2.1.1 标签分组功能升级方案.md`，分阶段实现标签分组能力。

标签功能是增量升级，不能破坏已有的全部功能。

当前版本不做标签筛选条。

本开发计划已经执行完成。

- 阶段 2：侧边栏标签分组显示，已完成并通过手动验证。
- 阶段 3：标签菜单、创建与编辑标签，已完成并通过手动验证。

```powershell

```

均已通过。

## 阶段 1：标签解析、数据字段与文档保存

### 目标

让任务能够识别自己的标签。

本阶段先完成底层标签能力，不要求侧边栏出现最终分组效果。

### 开发内容

创建或编辑任务时，用户可以在输入内容开头写标签。

示例：

```text
#项目A #阶段1 写测试用例
```

Task Flow 识别前两个开头标签：

- `#项目A` 是主标签。
- `#阶段1` 是子标签。
- `写测试用例` 是任务名称。

写入 Markdown 文档时，保存为任务名称在前、标签在后。

示例：

```markdown
- [ ] 写测试用例 #项目A #阶段1
```

从 Markdown 文档读取任务时，再从任务名称末尾解析标签，并写入任务数据结构中的标签字段。

这些标签字段只属于插件运行时任务数据，不是写入 Markdown 的新字段。

如果输入内容只有标签，没有任务名称，则不能创建任务，并提示 `任务名称不能为空`。

如果输入：

```text
#项目A #阶段1 #你好
```

则 `#你好` 作为任务名称处理。

### 手动验证

创建普通任务，确认文档保存和原来一致。

创建 `#项目A 写测试用例`，确认文档保存为：

```markdown
- [ ] 写测试用例 #项目A
```

创建 `#项目A #阶段1 写接口测试`，确认文档保存为：

```markdown
- [ ] 写接口测试 #项目A #阶段1
```

创建 `#项目A #阶段1 #你好`，确认文档保存为：

```markdown
- [ ] #你好 #项目A #阶段1
```

只输入 `#项目A #阶段1`，确认不能创建，并提示 `任务名称不能为空`。

打开已有带标签的 Markdown 任务，确认普通任务功能不受影响。

## 阶段 2：侧边栏标签分组显示

### 目标

让用户在侧边栏中看到标签分组效果。

### 开发内容

侧边栏保持现有任务卡片流，不改成纯文字树。

没有标签的任务直接显示在最上面，不显示 `未分类` 标题。

主标签显示为浅绿色标签胶囊，作为任务组标题。

同一主标签下，同一子标签拥有多个任务时，该子标签显示为子分组。

子标签标题前显示小圆点。

子标签胶囊和它下面的任务卡片一起向右缩进，并且左边缘对齐。

同一主标签下，只有一个任务使用的第二标签不形成子分组，而是显示在任务卡片内部。

卡片内标签胶囊放在任务类型文字后面，与任务类型处于同一行。

侧边栏显示任务名称时，隐藏 Markdown 末尾用于分组的标签。

### 手动验证

无标签任务显示在最上面，并且没有 `未分类` 标题。

多个 `#项目A` 任务显示在 `#项目A` 主标签下面。

多个 `#项目A #阶段1` 任务形成 `#阶段1` 子分组。

只有一个 `#项目A #阶段2` 任务时，`#阶段2` 显示在任务卡片内部，不形成子分组。

子标签胶囊和子标签下的任务卡片左边缘对齐。

任务名称中不重复显示已经用于分组的标签。

## 阶段 3：标签菜单、创建与编辑标签

### 目标

让用户可以通过标签胶囊操作标签任务。

### 开发内容

标签胶囊作为标签操作入口。

桌面端右键标签胶囊，弹出标签操作菜单。

手机端长按标签胶囊，弹出标签操作菜单。

弹出菜单不是因为第一版只有“新增标签任务”也非要多一步，而是为了给后续标签相关操作预留统一入口。

目前这里只实现和标签创建、标签编辑相关的菜单能力。

点击主标签的 `新增标签任务` 时，创建输入框预填当前主标签。

示例：

```text
#项目A 
```

点击子标签的 `新增标签任务` 时，创建输入框预填当前主标签和子标签。

示例：

```text
#项目A #阶段1 
```

标签组名称通过标签胶囊菜单里的 `编辑标签` 批量修改。

编辑主标签时，批量修改所有使用该主标签的任务的第一个标签。

编辑子标签时，只修改当前主标签下使用该子标签的任务的第二个标签。

编辑标签输入框只显示当前标签本身。

输入为空时，不修改。

输入包含空格时，确认后弹出提示，不修改。

输入不以 `#` 开头时，确认后弹出提示，不修改。

### 手动验证

桌面端右键主标签和子标签，都能打开标签操作菜单。

从主标签新增任务时，输入框预填 `#主标签 `。

从子标签新增任务时，输入框预填 `#主标签 #子标签 `。

提交后，Markdown 文档仍保存为任务名称在前、标签在后。

编辑主标签，确认所有使用该主标签的任务都被修改。

编辑子标签，确认只影响当前主标签下的对应子标签。

不同主标签下的同名子标签不会被误改。

输入为空、不以 `#` 开头、包含空格时，确认不会写入非法标签。

## 阶段 4：排序与拖拽

### 目标

让标签分组顺序、任务顺序和文档区顺序稳定一致。

### 开发内容

标签排序和任务拖拽都通过任务数据结构中的顺序实现。

侧边栏显示和 Markdown 文档写回都以同一份任务顺序为准。

实现时不直接手写调整 Markdown 文本顺序。

默认情况下，主标签按当前任务范围内第一次出现的顺序排列。

子标签按当前主标签下第一次出现的顺序排列。

组内任务按照任务数据结构中的顺序显示。

标签胶囊菜单中提供 `排序` 功能。

标签排序当前版本已经实现。

用户调整过标签顺序后，主标签和子标签按照用户调整后的顺序显示。

新出现的主标签追加到当前主标签排序后面。

新形成的子标签追加到当前主标签下已有子标签排序后面。

每个主标签内部先显示普通任务，再显示已经形成的子标签分组。

主标签普通任务包括只有主标签的任务，也包括有第二标签但未形成子分组的任务。

只有实际形成子分组的二级标签，才进入标签排序功能。

任务拖拽仍然遵守现有任务排序规则。

子任务只能在父任务内部拖拽排序。

带标签的任务只能在当前标签组内部拖拽。

无标签任务只能在无标签任务区域内拖拽。

任务不能通过跨标签拖拽改变标签归属。

### 手动验证

没有手动排序时，主标签按首次出现顺序显示。

标签排序菜单可以调整主标签顺序。

标签排序菜单可以调整已经形成子分组的子标签顺序。

新出现的主标签追加到已有排序后面。

新形成的子标签追加到当前主标签下已有子标签排序后面。

主标签普通任务排在子标签分组前面。

未形成子分组的二级标签不进入标签排序功能。

带标签任务只能在当前标签组内部拖拽。

无标签任务只能在无标签任务区域内拖拽。

子任务仍然只能在父任务内部拖拽。

拖拽或标签排序后，Markdown 文档中的任务行顺序和侧边栏顺序一致。

## 阶段 5：移动端与完整回归

### 目标

确认手机端规则一致，并且已有功能没有被标签功能破坏。

### 开发内容

移动端和桌面端规则保持一致。

除了标签菜单触发方式不同，其余标签显示、分组、排序、创建、编辑规则都一致。

当前版本不为手机端额外改造顶部时间选择器、日期列表、底部操作栏和创建页结构。

### 手动验证

手机端长按主标签胶囊可以打开标签菜单。

手机端长按子标签胶囊可以打开标签菜单。

手机端标签分组显示和桌面端规则一致。

手机端子标签缩进不破坏任务卡片布局。

手机端未形成子分组的第二标签显示在任务类型文字后面。

手机端顶部时间选择器、日期列表、底部操作栏和创建页结构保持稳定。

无标签任务创建、删除、修改、状态切换保持原有行为。

子任务、拖拽排序、跳转定位、同步、外部文档操作保持原有行为。

最后至少运行：

```powershell
npm.cmd run build
npm.cmd test
```
