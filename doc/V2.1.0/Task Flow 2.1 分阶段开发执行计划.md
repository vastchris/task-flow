# Task Flow 2.1 分阶段开发执行计划

## 1. 计划定位

本文决定 Task Flow 2.1 按什么顺序开发。

功能的完整业务规则以 `doc/2.1/Task Flow 2.1 基础模型.md` 为准。各阶段详细规则以对应 Stage 方案文档为准。本计划只说明：

- 每个阶段解决什么问题
- 开始阶段前必须具备什么
- 本阶段必须完成什么
- 本阶段明确不做什么
- 本阶段不能破坏什么
- 如何判断本阶段真正完成

当前完成情况以 `doc/2.1/Task Flow 2.1 开发进度.md` 为准。

## 2. 与 2.0 的关系

2.1 在 2.0 的代码基础上继续开发。代码目录同为 `src/v2/`。

2.0 阶段 1-3（插件基础、UI 框架、数据基础）已通过验收，2.1 继承这些成果。

从阶段 4 开始，所有功能规则以 2.1 方案为准。核心差异：

- **取消投影概念**：所有显示的任务都是真实任务记录
- **父任务自动同步**：子任务出现在某日期时，父任务必须作为真实任务同时出现
- **状态规则统一**：Week/Day 父任务按子任务身份组全局汇总，Day 父任务显示正常状态圆圈 + 总进度按钮

2.0 的阶段 4 代码（`v2Created.ts`）基于旧模型编写，在 2.1 中重写。

## 3. 总体开发顺序

第一大部分完成侧边栏任务管理器：

1. 插件和日期基础（继承 2.0）
2. UI 框架（继承 2.0）
3. V2 数据基础（继承 2.0）
4. created — 按 `doc/2.1/Stage 4 创建-方案.md`
5. deleted — 按 `doc/2.1/Stage 5 删除-方案.md`
6. modified — 按 `doc/2.1/Stage 6 修改-方案.md`
7. 常规状态变更 — 按 `doc/2.1/Stage 7-8 状态-方案.md` 第 7.1 节
8. 全局状态重算 — 按 `doc/2.1/Stage 7-8 状态-方案.md` 第 7.2 节
9. 第一大部分总验收

第二大部分完成 Markdown 文档投影，另行制定计划。

阶段顺序不能随意交换：

- created 先建立完整关系
- deleted 才能基于完整关系安全移除
- modified 再基于稳定的创建和删除规则改变已有结构
- 常规状态变更必须建立在稳定任务结构上
- 全局状态重算必须建立在结构操作和常规状态规则都完成之后

## 4. 全阶段共同规则

### 4.1 阶段边界

一个阶段只实现本阶段负责的能力。

如果为了验证 UI 提前展示后续功能，只能实现显示或临时入口，不能提前引入后续阶段的数据修改规则。

阶段外功能必须记录为后续任务，不能顺手混入当前实现。

### 4.2 数据操作

所有结构操作必须通过一次完整事务完成：

1. 读取操作需要的完整旧数据
2. 检查操作是否合法
3. 计算全部数据变化
4. 一次性保存最终结果
5. 保存成功后刷新侧边栏

失败、取消或冲突时不能留下部分结果。

### 4.3 身份和关系

任何阶段都不能用任务名称判断任务身份。

任务身份只看 sourceWeekTaskId → sourceDayTaskId → id 链。

任何关系变化都必须同时维护正向和反向关系。

任何阶段完成后都不能存在：

- 悬空任务 ID
- 无效 parentId
- 无效 sourceWeekTaskId
- 无效 sourceDayTaskId
- 超过两层的父子结构
- 同一关系只更新一侧

### 4.4 阶段验收

每个阶段都需要：

- 自动测试通过
- 构建通过
- 手动验证通过
- 更新开发进度
- 建立 Git 检查点

未经手动确认，不能把阶段标记为完成。

## 5. 阶段 1-3（继承 2.0，已完成）

阶段 1：插件基础与月文档识别
阶段 2：Week、Day 与侧边栏框架
阶段 3：V2 数据基础与真实只读显示

这三个阶段的成果直接继承。数据结构和存储层（v2Schema、v2Store、v2Normalize、v2Id、v2Defaults）保持不变。

验收状态：测试通过、构建通过、手动验收通过。

## 6. 阶段 4：created 任务创建与关系建立

### 阶段目的

完成所有「产生新任务记录或新任务实例」的操作，按 2.1 数据模型建立完整关系。

### 前置条件

- 阶段 1-3 已完成
- `doc/2.1/Stage 4 创建-方案.md` 已确认

### 本阶段负责

按 `doc/2.1/Stage 4 创建-方案.md` 实现全部 6 类入口：

1. 创建任务（面板标题 +）
2. 创建子任务（任务行 +，含四个 Day 子区域）
3. Week 任务安排到 Day（右键菜单）
4. Day 任务建立 Week 来源（右键菜单）
5. Day 任务延续（右键菜单）
6. 通用规则和不变量校验

### 本阶段不做

- 不删除任务
- 不修改名称、顺序或层级
- 不移动 Day 日期
- 不实现正式状态系统

### 不能破坏

- 同名任务必须可以分别创建和安排
- 待安排视图不能重复显示已安排任务
- Day 不能显示未安排到当天的兄弟子任务
- 已创建关系不能只更新一侧
- 独立任务非未开始时不能变为父任务
- 延续链上任务添加到 Week 不能破坏延续链

### 完成标准

- 所有创建入口使用真实数据
- 创建后侧边栏和数据立即一致
- 关系字段完整且无重复实例
- 独立任务拒绝条件生效
- 自动测试通过、构建通过
- 阶段 4 手动验收通过

## 7. 阶段 5：deleted 任务删除与关系修复

### 阶段目的

完成所有「移除任务记录及其实例」的操作，清理所有引用关系。

### 前置条件

- 阶段 4 已完成
- `doc/2.1/Stage 5 删除-方案.md` 已确认

### 本阶段负责

按 `doc/2.1/Stage 5 删除-方案.md` 实现：

1. Week 区域删除：父任务（级联删子+Day实例）、子任务、独立任务
2. Day 区域删除：Week来源任务（删实例不删源）、Day创建任务、延续根任务（最近实例升级为新根）、延续实例（非根）
3. 删除确认弹窗（四等级：简单/父任务级联/跨区域级联/延续根继承）
4. 空父处理（三场景区分）
5. 删除前工作记录冲突检测：保留任务，或解除绑定并删除

### 本阶段不做

- 不创建任务
- 不修改名称、排序或日期
- 不实现状态系统

### 不能破坏

- 删除 Day 实例不能删除 Week 源任务
- 删除延续实例不能断掉延续链
- 工作记录正文不能自动删除；解除绑定只将 `tasklog::` 改为 `tasklog-removed::`
- 关系字段必须双向同步清理
- 冲突出现时删除尚未执行，暂停等待用户选择

### 完成标准

- 所有删除入口使用真实数据
- 删除后侧边栏和数据立即一致
- 关系引用全部清理干净
- 延续链继承正确
- 弹窗内容准确反映连锁影响
- 自动测试通过、构建通过
- 手动验收通过

## 8. 阶段 6：modified 名称、排序和日期移动

### 阶段目的

完成改名（沿身份链同步）、排序（顺序数组操作）和日期移动（仅 todo 任务）三类修改操作。

### 前置条件

- 阶段 5 已完成
- `doc/2.1/Stage 6 修改-方案.md` 已确认

### 本阶段负责

按 `doc/2.1/Stage 6 修改-方案.md` 实现：

1. 改名：有链任务全链同步+弹窗确认、无链任务直接改、空名称拒绝
2. 排序：顶层排序、子任务排序（不支持跨父拖拽）
3. 日期移动：六种身份（Week来源父/子/独立 + Day创建父/子/独立），父移子必跟、有不满足者整体拒绝、子单独移自动补父

### 本阶段不做

- 不创建或删除任务
- 不调整层级（父子结构建立后固定）
- 不实现状态系统

### 不能破坏

- 身份链名称必须一致
- 父子关系不能通过排序或移动改变
- 日期移动后任务仍在同周范围内
- 子任务不能单独出现在 Day 中
- 延续链上的任务不能被移动（被 status 拒绝条件挡住）

### 完成标准

- 改名沿链同步且有弹窗确认
- 排序正确操作顺序数组
- 日期移动按身份正确执行
- 自动测试通过、构建通过
- 手动验收通过

## 9. 阶段 7：常规任务状态变更

### 阶段目的

实现侧边栏圆圈点击和 Markdown tasklog 变化驱动的任务状态更新，包含非父 Day 任务状态、跨日期同源合并、父任务汇总、Week 同步。

### 前置条件

- 阶段 6 已完成
- `doc/2.1/Stage 7-8 状态-方案.md` 第 7.1 节已确认

### 本阶段负责

按 `doc/2.1/Stage 7-8 状态-方案.md` 第 7.1 节实现：

1. 圆圈点击：红圈复制工作记录模板、蓝圈 `[/]→[x]`、绿勾 `[x]→[/]`、灰圈/蓝角标不可点击
2. tasklog 变化：新增 `[ ]→[/]`、删除 `[/]`或`[x]→[ ]`
3. 非父 Day 任务状态规则（7.1.2）
4. 跨日期同源 Day 非父任务状态合并（7.1.3）
5. Day 父任务状态与进度汇总（7.1.4）
6. Week / Day 任务状态同步（7.1.6）
7. 常规状态更新总流程（7.1.7）
8. tasklog 新增/删除在一轮中批量计算；活动编辑器只替换状态行，避免与 Obsidian Sync 双写
9. Day 移动/延续菜单权限：全 todo 只移动、全 doing 只延续、混合或含 done 均隐藏并提示

### 本阶段不做

- 不处理结构变化后的全局重算（阶段 8）
- 不创建、删除或修改任务结构

### 不能破坏

- 父任务和 Week 任务圆圈不可点击
- `[x]` 必须来自用户点击完成，tasklog 只让任务进入 `[/]`
- 同源组合并状态正确反映跨日期整体进度
- Week 状态始终从 Day 派生

### 完成标准

- 圆圈点击行为正确
- tasklog 防抖扫描触发状态更新正确
- 活动编辑器状态写回不触发 Obsidian Sync 外部修改冲突
- 同源组合并和父任务汇总结果正确
- `[ ] ✅` 与 `[/] ✅` 均正确显示并写回 Markdown
- 父任务进度按子任务身份组去重
- 侧边栏和 Markdown 状态一致
- 自动测试通过、构建通过
- 手动验收通过

## 10. 阶段 8：结构变化后的全局状态重算

### 阶段目的

在阶段 4/5/6 每次结构更新完成后自动触发全局状态重算，批量更新所有受影响任务的状态。

### 前置条件

- 阶段 7 已完成
- `doc/2.1/Stage 7-8 状态-方案.md` 第 7.2 节已确认

### 本阶段负责

按 `doc/2.1/Stage 7-8 状态-方案.md` 第 7.2 节实现：

1. 结构更新完成后自动触发
2. 扫描当前 Markdown tasklog 集合
3. 检测已删除的 tasklog 并更新对应任务状态
4. 全局重新计算所有同源组和父任务状态
5. Week 侧全量同步
6. 更新 confirmedTaskLogs
7. 写回发生变化的 Markdown 任务行

### 本阶段不做

- 不改变任务结构
- 不处理圆圈点击或 tasklog 变化的局部联动（那是阶段 7 的职责）

### 不能破坏

- 全局重算使用与阶段 7 相同的状态规则
- 只写回状态发生变化的任务行，不动其他 Markdown 内容
- tasklog-removed:: 不被纳入扫描

### 完成标准

- 任何结构操作完成后状态自动重新计算
- 全局重算结果与逐个手动计算一致
- 自动测试通过、构建通过
- 手动验收通过

对阶段 1-8 做完整的端到端验收，确保侧边栏任务管理器在真实月文档中可用。

### 前置条件

- 阶段 8 已完成

### 本阶段负责

### 完成标准

- 全部功能在真实 Obsidian 环境中正常工作
- 所有边界情况处理正确
- 侧边栏流畅不卡顿
- 无已知 bug
