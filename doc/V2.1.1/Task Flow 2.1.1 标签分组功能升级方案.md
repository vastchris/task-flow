# Task Flow 2.1.1 标签分组功能升级方案

## 目标

为任务增加基于 Obsidian 原生标签语法的分类能力。

用户可以在任务内容中使用 `#标签` 形式标记任务，例如：

```markdown
- [ ] 任务A #项目A #阶段1
```

Task Flow 在侧边栏中识别这些标签，并按标签结构组织任务显示。

## 完成状态

该功能已经完成开发，并经过自动测试和用户手动验证。

已完成能力：

- 创建、编辑任务时识别开头标签，写入 Markdown 时保存为任务名称在前、标签在后。
- 侧边栏按主标签、子标签分组显示。
- 未形成子分组的二级标签显示在任务卡片内部。
- 标签胶囊菜单支持新增标签任务、编辑标签、标签排序。
- 手机端长按标签胶囊打开菜单，并提供震动反馈。
- 标签排序保存后只刷新当前任务区域，并保留任务列表滚动位置。
- 当前版本不做标签筛选条。

最终验证：

```powershell
npm.cmd run build
npm.cmd test
```

均已通过。

## 标签分组规则

### 标签存储方式

标签直接写入任务行文本，使用 Obsidian 原生支持的 `#xxx` 语法。

示例：

```markdown
- [ ] 任务A #项目A
- [ ] 任务B #项目A #阶段1
```

Markdown 任务行基础格式不变，不新增隐藏字段。

实现时在任务数据结构中增加可选 `tags` 字段，用于插件运行时分组、显示和排序；该字段不写入 Markdown 文档。

### 第一版只支持 2 级标签分组

Task Flow 第一版标签分组只解释前两个标签：

- 第一个标签作为主标签。
- 第二个标签作为子标签。
- 第三个及后续标签只作为普通 Obsidian 标签保留，不参与 Task Flow 分组。

示例：

```markdown
- [ ] 任务A #项目A #阶段1 #测试
```

Task Flow 分组时只使用：

```text
#项目A / #阶段1
```

`#测试` 仍保留在任务文本中，可被 Obsidian 原生标签能力识别，但不参与 Task Flow 第一版标签分组。

### 子标签自动分组规则

侧边栏显示时，同一主标签下，同一子标签拥有多个任务时，才显示为子分组。

如果某个子标签下只有一个任务，则不单独显示子分组，该任务直接显示在主标签下，并保留子标签提示。

示例任务：

```markdown
- [ ] 任务1 #项目A
- [ ] 任务2 #项目A #阶段2
- [ ] 任务3 #项目A #阶段1
- [ ] 任务4 #项目A #阶段1
```

侧边栏期望显示：

```text
#项目A
  任务1
  任务2 #阶段2

  #阶段1
    任务3
    任务4
```

其中：

- `任务1` 没有子标签，直接显示在 `#项目A` 下。
- `任务2` 的 `#阶段2` 只有一个任务，不单独成组。
- `任务3` 和 `任务4` 共享 `#阶段1`，因此显示 `#阶段1` 子分组。

## 侧边栏显示方式

侧边栏保持现有任务卡片流，不改成纯文字树。

标签作为任务卡片之间的分组标题出现，用于把任务按标签分段显示。

### 主标签显示

主标签显示为浅绿色标签胶囊，左对齐显示在任务列表区域中。

示例：

```text
#小论文
[任务111]
[任务222]
[任务333]

#测试
[任务111]
[任务222]
[任务333]
```

主标签不折叠，不显示展开或收起按钮。

主标签下的任务卡片保持现有任务卡片样式和正常宽度。

已经作为主标签标题显示的标签，不在任务卡片内重复显示。

### 子标签显示

同一主标签下，同一子标签拥有多个任务时，该子标签显示为子分组。

子分组不折叠，不显示展开或收起按钮。

子标签标题前显示一个小圆点，用于提示这是子分组。

子标签胶囊和它下面的任务卡片一起向右缩进，并且左边缘对齐。

子标签下的任务卡片保持现有卡片样式，只是整体缩进，宽度随缩进变窄。

示例：

```text
#小论文
[任务111]
[任务222]
[任务333]

  · #self
  [任务111]
  [任务222]
  [任务333]
```

已经作为子分组标题显示的标签，不在该子分组任务卡片内重复显示。

### 未形成子分组的子标签显示

如果某个第二标签在同一主标签下只有一个任务使用，则不单独显示为子分组。

该第二标签显示在任务卡片内部，作为任务的标签胶囊。

卡片内标签胶囊放在任务类型文字后面，与任务类型处于同一行、行内对齐。

示例：

```text
#小论文
[ 111          + ]
[ 日任务  #代办 ]
```

其中 `#代办` 是第二标签，但没有形成子分组，因此显示在任务卡片内部。

## 创建任务时如何添加标签

普通创建入口仍然保留。

用户可以在创建输入框开头直接输入标签。

只有任务内容开头连续出现的 `#标签` 会被识别为 Task Flow 标签。

标签之间必须使用空格分隔，标签和任务名称之间也必须使用空格分隔。

示例：

```text
#项目A #阶段1 写测试用例
```

其中：

- `#项目A` 作为主标签。
- `#阶段1` 作为子标签。
- `写测试用例` 作为任务名称显示。

写入 Markdown 文档时，标签放在任务名称末尾。

示例：

```markdown
- [ ] 写测试用例 #项目A #阶段1
```

因为第一版只支持 2 级标签，所以最多只识别开头的前两个标签。

示例：

```text
#项目A #阶段1 #你好
```

其中：

- `#项目A` 作为主标签。
- `#阶段1` 作为子标签。
- `#你好` 作为任务名称显示。

如果只有标签，没有任务名称，则不能创建任务，并提示 `任务名称不能为空`。

示例：

```text
#项目A #阶段1
```

侧边栏显示任务名称时，隐藏末尾用于分组的标签。

标签胶囊作为标签操作入口：

- 桌面端右键标签胶囊，弹出标签操作菜单。
- 手机端长按标签胶囊，弹出标签操作菜单。

弹出菜单不是因为第一版只有“新增标签任务”也非要多一步，而是为了给后续标签相关操作预留统一入口。

当前版本菜单中已经实现 `新增标签任务`、`编辑标签` 和 `排序`。

点击主标签的 `新增标签任务` 时，创建输入框预填当前主标签。

示例：

```text
#项目A 
```

点击子标签的 `新增标签任务` 时，创建输入框预填当前主标签和子标签。

示例：

```text
#项目A #阶段1 
```

最终任务名称和标签以用户提交时输入框中的实际文本解析结果为准。

写入 Markdown 文档时，Task Flow 将识别出的标签放到任务名称末尾保存。

## 创建后如何修改标签

### 修改单个任务的标签

单个任务的标签通过现有任务编辑能力修改。

用户可以直接编辑任务的标签。

编辑输入时，可以使用开头标签表达标签归属。

示例：

```text
#项目A #阶段1 写测试用例
```

修改为：

```text
#项目A #阶段2 写测试用例
```

只影响当前这一条任务。

写回 Markdown 文档时，仍然保存为任务名称在前、标签在后。

### 批量修改标签名称

标签组的名称通过标签胶囊菜单里的 `编辑标签` 批量修改。

编辑主标签时，批量修改所有使用该主标签的任务的第一个标签。

编辑子标签时，只修改当前主标签下使用该子标签的任务的第二个标签。

示例：

```markdown
- [ ] 任务1 #项目A #阶段1
- [ ] 任务2 #项目A #阶段1
- [ ] 任务3 #项目B #阶段1
```

在 `#项目A` 下编辑子标签 `#阶段1` 为 `#阶段2` 后：

```markdown
- [ ] 任务1 #项目A #阶段2
- [ ] 任务2 #项目A #阶段2
- [ ] 任务3 #项目B #阶段1
```

`#项目B #阶段1` 不受影响。

### 编辑标签输入框

编辑标签时，输入框只显示当前标签本身。

编辑主标签时，输入框显示：

```text
#项目A
```

编辑子标签时，输入框只显示：

```text
#阶段1
```

输入框不显示完整标签路径。

### 编辑校验

输入为空时，不修改。

输入包含空格时，确认后弹出提示，不修改。

输入不以 `#` 开头时，确认后弹出提示，不修改。

## 文档区是否按标签显示

Markdown 文档本身不按标签重排。

文档仍然保持原来的时间结构。

文档区不写入标签分组标题，也不写入 `未分类` 标题。

标签排序会影响任务数据结构中的任务顺序。

侧边栏显示和 Markdown 文档写回都以同一份任务顺序为准。

Task Flow 不直接手写调整 Markdown 文本顺序，而是通过任务数据结构中的顺序实现文档区和侧边栏的一致。

文档中的任务行保存为任务名称在前、标签在后。

示例：

```markdown
- [ ] 写测试用例 #项目A #阶段1
- [ ] 整理需求 #项目A
- [ ] 修复问题 #项目B
```

创建或编辑时，Task Flow 可以识别用户输入的开头标签。

写入 Markdown 文档时，Task Flow 将识别出的标签放到任务名称末尾保存。

侧边栏显示任务名称时，隐藏末尾用于分组的标签，只显示真正的任务名称。

## 排序规则

### 基本原则

标签分组启用后，任务顺序由两部分共同决定：

- 标签组顺序。
- 组内任务顺序。

标签组顺序决定主标签和子标签分组的先后。

组内任务顺序决定同一个标签组内部的任务先后。

文档区不写入标签分组标题，但文档中的任务行顺序需要和侧边栏中的任务顺序保持一致。

这种一致性通过任务数据结构中的顺序实现。

标签排序和任务拖拽都应该更新任务数据结构中的顺序。

侧边栏显示和 Markdown 文档写回都以同一份任务顺序为准。

实现时不直接手写调整 Markdown 文本顺序。

### 无标签任务

没有标签的任务直接显示在最上面，不显示 `未分类` 标题。

无标签任务不进入任何标签组。

无标签任务之间按照任务数据结构中的顺序显示。

拖拽时，无标签任务只能在无标签任务区域内调整顺序。

无标签任务不能拖进标签组，标签任务也不能拖到无标签任务区域。

如果需要改变任务是否有标签，通过编辑任务标签完成。

### 默认排序

在用户没有手动调整标签排序之前，标签使用默认排序。

主标签默认按当前任务范围内第一次出现的顺序排列。

示例：

```markdown
- [ ] 任务1 #标签B
- [ ] 任务2 #标签A
- [ ] 任务3 #标签B
```

默认显示为：

```text
#标签B
  任务1
  任务3

#标签A
  任务2
```

因为 `#标签B` 先于 `#标签A` 出现。

子标签也使用同样的默认规则。

在当前主标签下，哪个子标签先出现，哪个子标签就排在前面。

组内任务不按标签名、拼音或创建时间重新排序，而是按照任务数据结构中的顺序显示。

### 手动标签排序

标签胶囊菜单中提供 `排序` 功能，当前版本已经实现。

默认排序只用于用户还没有手动调整标签顺序的情况。

用户调整过标签顺序后，主标签和子标签按照用户调整后的顺序显示。

新出现的主标签追加到当前主标签排序后面。

新形成的子标签追加到当前主标签下已有子标签排序后面。

标签排序只在当前任务范围内生效，不作为全局标签排序。

当前是日任务时，只影响当前日任务范围。

当前是周任务时，只影响当前周任务范围。

示例：

```text
#项目A
  #阶段D

#项目B
  #阶段1
```

用户可以在标签排序中调整主标签之间的顺序，也可以调整同一主标签下已经形成子分组的子标签顺序。

### 主标签内的显示顺序

每个主标签内部先显示普通任务，再显示已经形成的子标签分组。

主标签普通任务包括：

- 只有主标签的任务。
- 有第二标签但未形成子分组的任务。

有第二标签但未形成子分组的任务，第二标签显示在任务卡片内部。

这类任务和只有主标签的任务一起参与主标签普通任务区域内的排序。

同一主标签下，同一子标签拥有多个任务并形成子分组后，该子分组显示在主标签普通任务之后。

示例：

```text
#项目A
  任务1
  任务2 #阶段D

  · #阶段1
    任务3
    任务4
```

其中 `#阶段D` 没有形成子分组，因此显示在任务卡片内部，并留在主标签普通任务区域。

只有实际形成子分组的二级标签，才进入标签排序功能。

没有形成子分组的二级标签，不进入标签排序功能。

### 任务拖拽排序

标签分组启用后，任务拖拽仍然遵守现有任务排序规则。

子任务只能在父任务内部拖拽排序。

非子任务可以拖拽排序，但带标签的任务只能在当前标签组内部拖拽。

无标签任务只能在无标签任务区域内拖拽。

任务不能通过跨标签拖拽改变标签归属。

如果需要把任务移动到另一个标签组，通过编辑任务标签完成。

## 标签视图和现有时间视图的关系

不新增独立标签视图。

标签分组直接作用于现有日任务 / 周任务列表。

当前是日任务时，在当天任务内按标签组织。

当前是周任务时，在本周任务内按标签组织。

## 标签筛选条

标签筛选条用于筛选当前时间范围内的主标签任务。

点击某个主标签后，只显示当前时间范围内属于该主标签的任务。

点击 `全部` 后，恢复显示当前时间范围内的全部任务。

桌面端标签筛选条放在任务列表上方。

手机端标签筛选条放在底部操作栏上一行。

当前版本不做标签筛选条。

## 移动端影响范围

移动端和桌面端规则保持一致。

除了标签菜单触发方式不同，其余标签显示、分组、排序、创建、编辑规则都一致。

桌面端通过右键标签胶囊打开标签菜单。

手机端通过长按标签胶囊打开标签菜单。

长按成功触发菜单时，手机端提供震动反馈，用于提示用户已经按中标签胶囊。

当前版本不为手机端额外改造顶部时间选择器、日期列表、底部操作栏和创建页结构。

## 兼容与测试范围

标签功能是增量升级，不能破坏已有的全部功能。

已有无标签任务、任务创建、删除、修改、状态切换、子任务、拖拽排序、跳转定位、同步、外部文档操作、手机端布局和滚动等功能，都必须保持原有行为。

测试时需要同时覆盖标签任务和无标签任务。

标签任务重点验证标签识别、创建、显示、编辑、排序和菜单操作。

无标签任务重点验证原有任务流程不受影响。

桌面端和手机端都需要验证。
