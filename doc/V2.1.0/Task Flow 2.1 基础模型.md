# Task Flow 2.1 方案

## 1. 为什么做 2.1

2.0 的数据模型中存在「投影」概念——当 Week 子任务单独延续到某个日期，或 Day 子任务延续而其父任务未延续时，渲染层临时生成一个假父容器来保留层级。这个假容器不是真实任务记录，有特殊 ID、特殊删除规则、特殊状态行为。

投影的存在导致：

- 删除逻辑要区分「真任务删除」和「投影删除」
- 修改逻辑要判断「投影不能改变层级」
- 状态逻辑要处理「投影没有自己的状态」
- 整体数据模型不统一，操作和显示之间总有边界情况

2.1 取消投影概念。任何在一个日期中显示的任务，都是真实的任务记录。

---

## 2. 核心原则

### 2.1 没有投影

只要某个任务出现在某个日期中，它就是一条真实的 Day 任务记录，有真实 ID、有完整字段。不再存在渲染时拼出来的假条目。

子任务出现在某天 → 其父任务必须作为真实任务同时出现在该天。

### 2.2 任务身份

所有任务的类型由其来源关系和父子关系共同决定。身份是稳定的，不因出现在不同日期而改变。

---

## 3. 任务身份分类

### 3.1 Week 区域

Week 区域中的任务只有一个来源：在 Week 中直接创建。

| 代号 | 名称 | 判定 |
|------|------|------|
| WK1 | Week 父任务 | 有子任务（childIds 非空） |
| WK2 | Week 子任务 | 有父任务（parentId 非空） |
| WK3 | Week 独立任务 | 无父无子 |

### 3.2 Day 区域 — 以 day1 为例

Day 区域中的任务有两类来源：来自 Week，或在 Day 中直接创建。

**来自 Week 的任务**（sourceWeekTaskId 非空）：

| 代号 | 名称 | 判定 |
|------|------|------|
| W1 | Week 来源父任务 | source → WK1，有子任务 |
| W2 | Week 来源子任务 | source → WK2，有父任务 |
| W3 | Week 来源独立任务 | source → WK3，无父无子 |

**Day 中直接创建的任务**（sourceWeekTaskId / sourceDayTaskId 均为空）：

| 代号 | 名称 | 判定 |
|------|------|------|
| D1 | Day 父任务 | 有子任务 |
| D2 | Day 子任务 | 有父任务 |
| D3 | Day 独立任务 | 无父无子 |

**Day 延续的根任务**（sourceDayTaskId 为空，daytdayTaskIds 非空）：

| 代号 | 名称 | 判定 |
|------|------|------|
| D4 | Day 延续根父任务 | D1 被延续后成为 D4，有子任务 |
| D5 | Day 延续根子任务 | D2 被延续后成为 D5，有父任务 |
| D6 | Day 延续根独立任务 | D3 被延续后成为 D6，无父无子 |

### 3.3 Day 区域 — 以 day2 为例

day2 是 day1 的延续目标日期。任务来自两类延续：

**Week 来源任务的延续**：

| 代号 | 名称 | 判定 |
|------|------|------|
| W1延续 | Week 来源父任务延续 | source → WK1，真实的 Day 任务 |
| W2延续 | Week 来源子任务延续 | source → WK2 |
| W3延续 | Week 来源独立任务延续 | source → WK3 |

Week 来源任务的延续实例，sourceWeekTaskId 始终指向 Week 源。不因延续而变成 Day 来源。

**Day 创建任务的延续**：

| 代号 | 名称 | 判定 |
|------|------|------|
| D1延续 | Day 父任务延续实例 | source → D4 |
| D2延续 | Day 子任务延续实例 | source → D5 |
| D3延续 | Day 独立任务延续实例 | source → D6 |

Day 创建任务的延续实例，sourceDayTaskId 指向根任务（D4/D5/D6）。

## 4. 状态规则

### 4.1 父任务状态：全局汇总

父任务的状态只有一个来源：它旗下所有**子任务身份组**（不管在哪个日期）的全局汇总。

子任务身份组表示同一项实际工作的全部实例：

- 只存在一个日期的子任务：该实例单独构成一个身份组
- 来自同一个 Week 子任务的多个 Day 实例：合并为一个身份组
- 同一个 Day 子任务延续到多个日期：根任务和全部延续实例合并为一个身份组

父任务状态和进度按身份组计算，不按 Day 实例数量计算。

- 所有子任务未开始 → 父任务未开始
- 部分子任务进行中或完成 → 父任务进行中
- 所有子任务完成 → 父任务已完成

同一个逻辑父任务在任何日期看到的状态都一样。状态不分日期计算。

例如 WK1 旗下 3 个子任务，分布在不同日期，WK1 的状态始终由这 3 个子任务的全局汇总决定。day1 的 W1、day2 的 W1延续，显示的状态一致。

D4（Day 延续根父任务）同理。旗下子任务可能分散在 day1、day2、day3，父状态看全部。

### 4.2 状态圆圈显示

**Week 区域**：

- 父任务（WK1）→ 正常颜色圆圈，显示汇总状态
  - 未开始 = 红色空心圆
  - 进行中 = 蓝色描边圆圈、浅蓝填充
  - 已完成 = 绿色圆形、白色勾
- 子任务和独立任务（WK2、WK3）→ 正常颜色圆圈，显示自身状态

**Day 区域**：

- 所有父任务（W1、D1、D4 及它们的延续实例）→ 正常颜色圆圈，显示身份组汇总状态
- 所有子任务和独立任务（W2、W3、D2、D3、D5、D6 及延续实例）→ 正常颜色圆圈，显示自身状态
- 跨日期同源任务若已在其他日期完成：
  - `[ ] ✅` → 灰圈 + 绿色小勾角标
  - `[/] ✅` → 蓝圈 + 绿色小勾角标

父任务圆圈不可点击，只显示身份组汇总状态。特殊状态圆圈也不可点击。

### 4.3 Day 父任务总进度

Week 和 Day 区域的父任务使用一致的状态与进度语义。Day 父任务旁显示总进度按钮，格式为「已完成身份组数 / 身份组总数」，例如 `总进度 2/5`。

总进度同样是全局汇总——同一个子任务即使出现在多个日期，也只计一次。

进度详情按「进行中 / 未开始 / 已完成」分组。跨日期身份组只显示一行：主日期取最新日期，并列出其他保留日期。

### 4.4 Day 操作权限

- 延续只适用于当前日期全部处于 `doing` 的 Day 任务
- 移动只适用于当前日期全部处于 `todo` 的 Day 任务
- Day 父任务按当前日期旗下全部子任务判断
- 状态混合或包含 `done` 时，移动和延续入口均不显示
- 多选使用同一规则；父任务展开为当前日期下的实际子任务后判断
- Week 区域不提供 Day 移动或延续入口

### 4.5 tasklog 与多日未开始

同源任务的全部 tasklog 被删除后，所有实例可以恢复为 `todo`。日期实例继续保留，不自动删除；父任务状态和进度仍将这些实例合并为一个身份组计算。

---

## 5. 数据关系

每个任务记录包含以下关系字段，创建操作保证正反两侧同步：

**父子关系**：

- 父任务记录所有子任务 ID
- 子任务记录父任务 ID

**Week→Day 来源**：

- Week 任务记录它生成了哪些 Day 实例
- Day 实例记录它来自哪个 Week 任务

**Day→Day 延续**：

- 根任务记录它有哪些延续实例
- 延续实例记录它延续自哪个根任务

---

## 6. 数据结构

### 6.1 顶层结构

数据存储在月文档的 `data.json` 中，顶层结构如下：

```json
{
  "version": 2,
  "updatedAt": "2026-06-10T...",
  "files": {
    "2026.6.md": {
      "tasks": { ... },
      "weeks": { ... },
      "confirmedTaskLogs": { "taskIds": [] }
    }
  }
}
```

一个 `files` 条目对应一个月文档。`tasks` 是所有任务记录的字典，`weeks` 是按周键组织的 Week/Day 数据。

### 6.2 任务记录（TaskRecord）

`tasks` 是一个字典，key 为任务 ID，value 为任务记录：

```json
{
  "tf-w-0001": {
    "id": "tf-w-0001",
    "area": "week",
    "areaKey": "2026.6.2-2026.6.8",
    "name": "需求分析",
    "status": "doing",
    "parentId": null,
    "childIds": ["tf-w-0002", "tf-w-0003"],
    "sourceWeekTaskId": null,
    "sourceDayTaskId": null,
    "weektdayTaskIds": ["tf-d-0010", "tf-d-0020"],
    "daytdayTaskIds": null
  }
}
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 唯一标识，Week 任务 `tf-w-` 前缀，Day 任务 `tf-d-` 前缀 |
| `area` | "week" \| "day" | 任务所属区域 |
| `areaKey` | string | 所属周键（如 `2026.6.2-2026.6.8`）或日期键（如 `2026.6.5`） |
| `name` | string | 任务名称，允许重名 |
| `status` | "todo" \| "doing" \| "done" | 自身状态 |
| `parentId` | string \| null | 父任务 ID，null 表示顶层任务 |
| `childIds` | string[] | 子任务 ID 列表，空数组表示无子任务 |
| `sourceWeekTaskId` | string \| null | Day 任务指向 Week 来源任务 |
| `sourceDayTaskId` | string \| null | 延续实例指向 Day 根任务 |
| `weektdayTaskIds` | string[] | Week 任务记录它生成了哪些 Day 实例（Week→Day 反向引用） |
| `daytdayTaskIds` | string[] \| null | Day 根任务记录它有哪些延续实例（Day→Day 反向引用），null 表示非根任务 |

### 6.3 顺序数组与嵌套结构

每个周和每个日期各有一个顺序数组，决定该区域内任务的显示排列。这是 2.1 与 2.0 的关键数据结构差异。

**TaskOrderItem 类型**：顺序数组中的每个元素有两种形态：

```
TaskOrderItem = string | { id: string, childIds: string[] }
```

- **独立任务/子任务**：直接用 ID 字符串表示，如 `"tf-w-0003"`
- **父任务**：用对象表示，`id` 是父任务自身 ID，`childIds` 是它在当前区域内显示的子任务 ID 列表（按显示顺序排列）

**Week 顺序数组示例**：

```json
"weekTaskIds": [
  "tf-w-0003",
  { "id": "tf-w-0001", "childIds": ["tf-w-0002", "tf-w-0004"] }
]
```

对应的显示顺序：

```
tf-w-0003          ← 独立任务
tf-w-0001          ← 父任务
  tf-w-0002        ← 子任务（缩进显示）
  tf-w-0004        ← 子任务（缩进显示）
```

**Day 顺序数组示例**：

```json
"dayTaskIds": [
  { "id": "tf-d-0010", "childIds": ["tf-d-0011"] },
  "tf-d-0020"
]
```

对应显示：

```
tf-d-0010          ← 父任务
  tf-d-0011        ← 子任务
tf-d-0020          ← 独立任务
```

### 6.4 为什么用嵌套结构

嵌套结构直接表达了父子层级关系，与 `tasks` 字典中的 `parentId`/`childIds` 字段形成双重保障。

关键约束：**顺序数组中的父子关系必须与 `tasks` 记录中的 `parentId`/`childIds` 保持一致。** 任何结构操作都必须同时维护两侧。

这个设计直接影响拖拽排序：拖拽一个任务意味着在顺序数组中移动它，如果是子任务则是在父任务的 `childIds` 数组中移动，如果是顶层任务则是在顶层数组中移动。嵌套结构使排序操作可以直接定位到正确的层级。

### 6.5 辅助操作

顺序数组提供四个基本操作：

- **展平**：将嵌套数组展开为纯 ID 列表，用于遍历
- **查找**：按 ID 在数组中定位元素及其索引
- **移除**：按 ID 从数组或子数组中移除，子数组变空时父条目自动降级为纯字符串
- **添加子任务**：将子任务 ID 添加到父条目的 `childIds` 中，若父条目当前为纯字符串则自动升级为对象

### 6.6 weeks 结构完整示例

```json
"weeks": {
  "2026.6.2-2026.6.8": {
    "weekTaskIds": [
      { "id": "tf-w-0001", "childIds": ["tf-w-0002"] },
      "tf-w-0003"
    ],
    "days": {
      "2026.6.5": {
        "dayTaskIds": [
          { "id": "tf-d-0010", "childIds": ["tf-d-0011"] }
        ]
      },
      "2026.6.6": {
        "dayTaskIds": []
      }
    }
  }
}
```

周键格式为 `起始日期-结束日期`。`weekTaskIds` 管理 Week 区域的显示顺序，`days` 下每个日期键对应一个 `DayData`，其中 `dayTaskIds` 管理该日期的显示顺序。

## 7. UI 布局

侧边栏保持两个主模块：

- 上方：周任务（Week）
- 下方：日任务（Day）

日任务底部：本周当前日期前未完成任务浮层（默认折叠）。

具体布局和交互规则继承 2.0 方案中的侧边栏整体布局章节，本文不重复。
